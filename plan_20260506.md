## HIGH

### TASK-001 — Write `bootstrap.sh`

- **File:** `bootstrap.sh` (vault root)
- **Purpose:** Single entry point fetched via `curl -fsSL https://raw.githubusercontent.com/growlf/opencode_audito/main/bootstrap.sh | bash`
- **Design:**

```
bootstrap.sh flow:
  1. Check prerequisites: git, jq (print install commands if missing)
  2. Offer to install opencode if missing (curl -fsSL https://opencode.ai/install | bash)
  3. Clone repo to $HOME/Audit (or $INSTALL_DIR env var)
     - If dir exists and is a git repo: git pull instead
     - If dir exists and is NOT a git repo: warn and exit
  4. cd $INSTALL_DIR
  5. Run setup/install.sh (handles: permissions, instructions, provider, skills, scaffolding, Obsidian plugin, workspace)
  6. Offer to install security tools (apt install lynis rkhunter fail2ban)
  7. Print success + next steps
```

**Key decisions:**
- Use `git clone` (not tarball) so `git pull` works for updates
- Honor `INSTALL_DIR` env var, default `$HOME/Audit`
- Non-interactive safe: `YES=1 ./bootstrap.sh` skips all prompts
- On re-run: `git pull` then re-run install.sh (idempotent)

**Self-critique:**
- **Trust issue:** `curl | bash` is powerful but opaque. Mitigation: add a `--dry-run` flag that prints what would happen without executing. Also document that users can `curl -fsSL URL -o bootstrap.sh && less bootstrap.sh && bash bootstrap.sh` to review first.
- **`git` dependency:** Most dev machines have git, but a `curl | tar` fallback would be more universal. Deferred — git is a reasonable requirement for a dev-security tool.
- **openocode dependency:** The bootstrap should offer to install opencode, not fail. Many users won't have it yet.

### TASK-005 — Strip bloated plugin artifacts from `setup/`

- `setup/.obsidian/plugins/opencode-obsidian/` contains full git history, node_modules, bun.lock, and source ts files — pulled from the growlf/opencode-obsidian repo
- These bloat the vault by ~40MB+ and are not needed — `install.sh` already downloads release artifacts (main.js, manifest.json, styles.css) at line 254-266
- **Action:** Replace the full plugin checkout with just the three release artifacts that install.sh actually uses, plus versions.json
- This keeps the vault lean and avoids shipping a second git repo inside the vault

## Self-Critique (Pre-Proposal)

- **Is the bootstrap worth it?** The current install flow requires `git clone && cd && bash setup/install.sh`. The bootstrap saves exactly one `git clone` and adds dependency checking. It's a quality-of-life improvement, not a feature change. For a tool that's installed once and updated rarely, the ROI is modest.
- **Version pinning risk:** `curl ... main/bootstrap.sh | bash` always fetches `main`. If main is broken, new users get broken installs. Mitigation: tag releases and document `curl ... v0.3.0/bootstrap.sh` as the stable URL.
- **Platform support:** The bootstrap currently targets Debian/Ubuntu for security tool install. It should detect distro and print appropriate commands, or skip if unsupported.
- **The `setup/.obsidian` bloat issue is real** — shipping a full git repo (opencode-obsidian) inside the vault is technically wrong and should be fixed regardless of the bootstrap effort.
