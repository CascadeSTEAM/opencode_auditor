# Enhancement Plan — Versioning & Tag Cleanup — 2026-05-06

**Premise:** Formalize a versioning scheme for the vault repo before pushing to GitHub. Clean up incorrect tags that don't represent code releases.

---

## Self-Critique

- **Dual numbering is confusing.** Git tags are v0.Y.Z, but `.vault-version` says `version=4.0`. These are unrelated and nothing cross-references them. The `.vault-version` file only served as a "has install.sh run?" flag — now removed entirely, since `mitigations/` existence serves that purpose.
- **Semver fits this project.** Even though it's not a library, users pin the bootstrap URL (`v1.0.0/bootstrap.sh`) and need to know what breaks. Major = workflow/infra breaking, Minor = new tools/features, Patch = fixes/docs.
- **Delete wrong tags before first push.** v0.4.0 tagged a plan file. v0.5.1 tagged archiving a plan. These are noise. On a fresh GitHub repo, they'd never exist — best to delete them now.

## HIGH

### VER-001 — Document Versioning Scheme

**Versioning scheme (Semver 2.0):**

```
vMAJOR.MINOR.PATCH
```

| Bump | When | Example |
|------|------|---------|
| MAJOR | Breaking changes to workflow, vault structure, install method, or AGENTS.md directives | v1.0.0 → v2.0.0 |
| MINOR | New features, optional tools, non-breaking doc additions, new skills | v0.5.0 → v0.6.0 |
| PATCH | Bug fixes, doc corrections, gitignore tweaks, non-functional refinements | v0.5.0 → v0.5.1 |

**Pre-1.0 convention:**
- v0.MINOR.PATCH (major is 0 during initial development)
- MINOR = feature/breaking changes
- PATCH = fixes, docs, non-functional

**What does NOT get a tag:**
- Plan files or plan archives (internal workflow)
- Moving files around (rename, archive)
- Single-doc edits without code/functional changes

**Release criteria for v1.0.0:**
- bootstrap.sh tested on Ubuntu, Fedora, macOS
- Install flow works clean (clone → install → opencode)
- All SOC2 docs finalized
- At least one full audit cycle completed

**File:** `VERSIONING.md` at repo root.

### VER-003 — Remove .vault-version Entirely

The `.vault-version` file existed for one purpose: "has install.sh been run?" The `mitigations/` directory serves that same purpose — it's created by `install.sh` and its existence is already checked by `startup.sh` (section 1). The `.vault-version` content (`version=4.0`, `setup_date`, `setup_complete`) was never read or compared by any code — purely cosmetic.

**Actions:**
- `git rm .vault-version setup/.vault-version` (remove from repo)
- Add `.vault-version` to `.gitignore` (safety net for stale copies)
- Remove vault-version check from `startup.sh` (mitigations/ check already exists)
- Remove vault-version creation from `install.sh`
- Update `docs/file-conventions.md`, `setup/INSTALL.md` references
- mitigations/ existence is now the install-complete marker
